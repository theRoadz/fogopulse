# Story 5-7.1: Fix LP Withdrawal Epoch Protection

## Story

As a liquidity provider, I want my withdrawal funds to be protected from being used in active epochs, so that the pool remains solvent and my withdrawal is guaranteed after cooldown.

## Status: in-progress

## Context

**Vulnerability:** The on-chain `process_withdrawal` instruction had no check preventing LP withdrawals while an epoch is active. An LP could drain pool reserves mid-epoch, leaving the pool insolvent for paying winners.

**Solution:** Added a `pending_withdrawal_shares` field to the Pool struct that tracks the aggregate LP shares pending withdrawal. At settlement rebalancing, the USDC value is computed dynamically from current share prices and subtracted from total reserves before the 50:50 split — effectively "reserving" the withdrawal funds. `process_withdrawal` is blocked during active epochs. A permissionless `crank_process_withdrawal` instruction enables the crank bot to auto-process pending withdrawals in the gap between settlement and next epoch creation.

## Acceptance Criteria

1. Given an active epoch, When I try to complete a withdrawal, Then the transaction fails with `WithdrawalBlockedDuringEpoch`
2. Given a pending withdrawal request, When the epoch settles and rebalances, Then the withdrawal amount is reserved (excluded from the 50:50 rebalance for the next epoch)
3. Given a pending withdrawal with elapsed cooldown and no active epoch, When the crank bot runs, Then it auto-processes the withdrawal before creating the next epoch
4. Given a pending withdrawal during an active epoch, When I view the LP dashboard, Then I see "Waiting for epoch settlement..." instead of the Complete button
5. Given the pool has `pending_withdrawal_shares > 0`, When `request_withdrawal` is called, Then `pending_withdrawal_shares` is incremented by the requested shares amount
6. Given `process_withdrawal` or `crank_process_withdrawal` completes, Then `pending_withdrawal_shares` is decremented by the processed shares

## Dev Notes

- Pool account size increased by 8 bytes (new `pending_withdrawal_shares: u64` field). Existing pools on testnet need reinitialization.
- Crank bot's `parsePoolAccount` offset updated to account for new field.
- `pending_withdrawal_shares` tracks shares (not USDC) to avoid drift between request and processing time. USDC value is computed dynamically at settlement.
- `settle_epoch` computes reserved USDC as `pending_withdrawal_shares * total_reserves / total_lp_shares` before the 50:50 rebalance.
- `crank_process_withdrawal` is a permissionless variant of `process_withdrawal` — enables the crank bot to auto-process withdrawals without needing user session delegation.
- Crank bot's `findPendingWithdrawals` filters by cooldown elapsed to avoid wasting transactions.
