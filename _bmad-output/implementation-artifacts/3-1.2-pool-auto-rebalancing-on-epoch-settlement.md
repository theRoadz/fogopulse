## User Story

**As a** trader
**I want** each new epoch to start with fair 50:50 pool reserves
**So that** I get fair CPMM pricing regardless of previous epoch outcomes

## Story Overview

Currently, pool reserves (`yes_reserves`, `no_reserves`) persist across epochs without rebalancing. This creates an unfair CPMM pricing advantage for traders betting against the previous epoch's popular direction.

**The Problem:**
- Epoch 0 ends with 45:55 (YES:NO) reserves
- Epoch 1 starts with the SAME 45:55 imbalance
- CPMM formula: `shares = amount Ă— opposite_reserves / same_reserves`
- DOWN bettors get 1.22Ă— advantage (55/45), UP bettors get 0.82Ă— disadvantage (45/55)

**The Solution:**
After epoch settlement, automatically rebalance reserves to 50:50 while preserving total liquidity.

```rust
let total = yes_reserves + no_reserves;
pool.yes_reserves = total / 2 + (total % 2);  // YES gets remainder
pool.no_reserves = total / 2;
```

### 2. Add Rebalancing Logic to settle_epoch
**File**: `anchor/programs/fogopulse/src/instructions/settle_epoch.rs`

Insert **after line 263** (`pool.active_epoch = None;`):

```rust
// ==========================================================
// AUTO-REBALANCE POOL RESERVES
// ==========================================================
// After settlement, rebalance reserves to 50:50 to ensure fair
// CPMM pricing for the next epoch. This prevents traders from
// exploiting imbalanced reserves where shares = amount * opposite / same
// creates unfair advantages for the scarce-side.

let yes_reserves_before = pool.yes_reserves;
let no_reserves_before = pool.no_reserves;

let total_reserves = pool.yes_reserves
    .checked_add(pool.no_reserves)
    .ok_or(FogoPulseError::Overflow)?;

let balanced_amount = total_reserves / 2;
let remainder = total_reserves % 2;

pool.yes_reserves = balanced_amount
    .checked_add(remainder)
    .ok_or(FogoPulseError::Overflow)?;
pool.no_reserves = balanced_amount;

emit!(PoolRebalanced {
    pool: pool.key(),
    epoch: epoch.key(),
    yes_reserves_before,
    no_reserves_before,
    yes_reserves_after: pool.yes_reserves,
    no_reserves_after: pool.no_reserves,
});

msg!(
    "Pool rebalanced: before=({}, {}), after=({}, {})",
    yes_reserves_before,
    no_reserves_before,
    pool.yes_reserves,
    pool.no_reserves
);
```

## Design Decisions

### Why `settle_epoch` (not `create_epoch`)?

1. **Atomic finalization** - All epoch cleanup in one transaction
2. **No oracle dependency** - Pure arithmetic on existing state
3. **Immediate consistency** - Pool balanced right after settlement
4. **Simpler testing** - Deterministic behavior

### Why not configurable per pool?

- Simplicity - Auto-rebalancing is a core invariant for fair CPMM
- No use case for persistent imbalance
- Can add `pool.auto_rebalance: bool` later if needed

### Security Considerations

| Concern | Mitigation |
|---------|------------|
| Manipulation attack | Rebalancing removes any advantage - desired behavior |
| Front-running | Rebalancing happens at settlement, before next epoch |
| Overflow | Protected by `checked_add` |
| Permissionless settle | No concern - rebalancing is deterministic |
