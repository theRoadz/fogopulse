# Story 1.8.1: Pyth Lazer Version Research Documentation

Status: done

## Summary

Research documentation for pyth-lazer-solana-contract version selection. This story documents findings from investigating why version 0.5.0 was chosen over 0.7.1+ and provides a migration path for future upgrades if needed.

## Research Date
2026-03-12

## Key Findings

### Why Version 0.5.0 Is Used

**The `cpi` module is only available in version 0.5.0:**

```toml
# Cargo.toml - Current working configuration
pyth-lazer-solana-contract = { version = "0.5.0", features = ["no-entrypoint", "cpi"] }
```

Version 0.7.1 does NOT export the `cpi` module. Attempting to use `pyth_lazer_solana_contract::cpi::verify_message()` with 0.7.1 results in a compilation error.

**Official Pyth examples also use 0.5.0:**
The [pyth-network/pyth-examples](https://github.com/pyth-network/pyth-examples/tree/main/lazer/solana-anchor) Solana Anchor example uses version 0.5.0 with `features = ["no-entrypoint", "cpi"]`.

### What Version 0.7.1+ Offers

Version 0.7.1 exports these modules (per [docs.rs](https://docs.rs/pyth-lazer-solana-contract/latest/pyth_lazer_solana_contract/)):
- `accounts` - Anchor account structs
- `instruction` - Instruction definitions including `VerifyMessage`
- `program` - Program ID and metadata
- `pyth_lazer_solana_contract` - Nested module
- `Ed25519SignatureOffsets` - Struct for Ed25519 verification setup
- `ed25519_program_args()` - Helper function for Ed25519 instruction data

**Notable absence:** No `cpi` module is exported.

### Two Integration Approaches

#### Approach 1: CPI Module (v0.5.0) - Current Implementation

```rust
use pyth_lazer_solana_contract::cpi::accounts::VerifyMessage as VerifyMessageAccounts;
use pyth_lazer_solana_contract::cpi::verify_message;

// Build CPI accounts
let cpi_accounts = VerifyMessageAccounts {
    payer: ctx.accounts.payer.to_account_info(),
    storage: ctx.accounts.pyth_storage.to_account_info(),
    treasury: ctx.accounts.pyth_treasury.to_account_info(),
    system_program: ctx.accounts.system_program.to_account_info(),
    instructions_sysvar: ctx.accounts.instructions_sysvar.to_account_info(),
};

let cpi_ctx = CpiContext::new(ctx.accounts.pyth_program.to_account_info(), cpi_accounts);
verify_message(cpi_ctx, pyth_message, ed25519_instruction_index.into(), signature_index.into())?;
```

**Pros:**
- Clean Anchor-idiomatic syntax
- Type-safe account validation
- Simpler code

**Cons:**
- Locked to version 0.5.0
- May have Anchor version compatibility issues
- Requires `anchor-lang = "0.31.1"` (not latest)

#### Approach 2: Raw invoke() (v0.7.1+) - Migration Path

```rust
use solana_program::program::invoke;
use solana_program::instruction::{Instruction as ProgramInstruction, AccountMeta};
use pyth_lazer_solana_contract::instruction::VerifyMessage;

// Build instruction manually
let verify_ix = ProgramInstruction::new_with_bytes(
    pyth_lazer_solana_contract::ID,
    &VerifyMessage {
        message_data: pyth_message.to_vec(),
        ed25519_instruction_index,
        signature_index,
    }.data(),
    vec![
        AccountMeta::new(*payer.key, true),
        AccountMeta::new_readonly(*pyth_storage.key, false),
        AccountMeta::new(*pyth_treasury.key, false),
        AccountMeta::new_readonly(*system_program.key, false),
        AccountMeta::new_readonly(*instructions_sysvar.key, false),
    ],
);

invoke(&verify_ix, accounts)?;
```

**Pros:**
- Works with latest version (0.7.1+)
- Version-agnostic pattern
- May allow upgrading to `anchor-lang = "0.32.x"`
- More control over instruction construction

**Cons:**
- More verbose code
- Manual account meta construction
- Less type safety (account order matters)

## Migration Guide (For Future Upgrades)

If upgrading to v0.7.1+ becomes necessary (e.g., for new features or Anchor version compatibility):

### Step 1: Update Cargo.toml

```toml
# Remove "cpi" feature, keep "no-entrypoint"
pyth-lazer-solana-contract = { version = "0.7.1", features = ["no-entrypoint"] }
```

### Step 2: Update Imports

```rust
// Remove these:
// use pyth_lazer_solana_contract::cpi::accounts::VerifyMessage as VerifyMessageAccounts;
// use pyth_lazer_solana_contract::cpi::verify_message;

// Add these:
use solana_program::program::invoke;
use solana_program::instruction::{Instruction as ProgramInstruction, AccountMeta};
use pyth_lazer_solana_contract::instruction::VerifyMessage;
```

### Step 3: Replace CPI Call

Replace the CPI pattern with raw invoke pattern (see Approach 2 above).

### Step 4: Test Ed25519 Verification

The verification flow remains the same:
1. Transaction includes Ed25519 instruction at index 0
2. Your instruction calls Pyth program to verify
3. Pyth program checks signature against instructions sysvar
4. On success, deserialize payload with `SolanaMessage` and `PayloadData`

## Version Compatibility Matrix

| Component | v0.5.0 (current) | v0.7.1+ (future) |
|-----------|------------------|------------------|
| anchor-lang | 0.31.1 | Potentially 0.32.x |
| CPI module | Yes | No |
| invoke() pattern | Yes | Yes |
| Ed25519SignatureOffsets | Yes | Yes |
| SolanaMessage | Yes | Yes |
| PayloadData | Yes | Yes |

## Decision Record

**Decision:** Stay on v0.5.0 with CPI module
**Date:** 2026-03-12
**Rationale:**
1. Current implementation works and is tested
2. Official Pyth examples use v0.5.0
3. No compelling features in v0.7.1 that we need
4. Migration would require refactoring with risk of regressions

**Revisit When:**
- Anchor 0.32.x becomes required for other dependencies
- Pyth releases v0.8+ with breaking changes or required features
- Security vulnerabilities discovered in v0.5.0
