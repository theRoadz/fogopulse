# Story 2.5.2: Price Chart Smooth Rendering

Status: in-progress

## Story

As a trader,
I want the price chart to update smoothly without flickering,
So that I can clearly see price movements without visual distractions.

## Problem Statement

The price chart time axis labels (HH:MM:SS AM/PM) and price line flicker during real-time updates. The `setVisibleRange()` function is called on every price update, forcing the chart to completely recalculate and redraw the time axis.

## Research Findings

### Lightweight Charts Library Limitations
- No built-in smooth animation for live data updates ([confirmed by maintainers](https://github.com/tradingview/lightweight-charts/issues/1465))
- `setVisibleRange()` is a heavy operation that forces complete time axis recalculation
- Library maintainers closed animation feature requests as "NOT_PLANNED"

### Industry Solutions
- `requestAnimationFrame` is recommended for batching updates with browser render cycles
- PolyChart (Polymarket Chrome extension) achieves smoothness via "hardware-accelerated canvas rendering and smooth 60fps interactions"
- [MDN documentation](https://developer.mozilla.org/en-US/docs/Web/API/Window/requestAnimationFrame) confirms rAF prevents flickering by syncing with browser paint cycles

### Root Cause Analysis
Current flow causing flicker:
1. Hermes delivers price update (~1/sec)
2. `usePriceHistory` creates new array → React re-render
3. `PriceChart` useEffect triggers → `scrollToRight()` called
4. `setVisibleRange()` forces complete time axis redraw → **FLICKER**

## Acceptance Criteria

1. **Given** real-time price updates, **When** the chart receives data, **Then** the time axis labels should not flicker
2. **Given** real-time price updates, **When** the chart receives data, **Then** the price line should update smoothly
3. **Given** the 60-second visible window requirement, **When** new data arrives, **Then** the window should shift smoothly without visual disruption
4. Chart should maintain 60-second fixed visible window (no change from 2.5.1)
5. Chart should auto-scroll when NEW second arrives (once per second, not on every price update)

## Technical Design

### Part 1: Skip Redundant setVisibleRange Calls

Most Pyth updates are price changes for the SAME timestamp (second). Only ~1 per second is a new timestamp requiring window shift.

```typescript
const lastScrolledTimeRef = useRef<number>(0)

const scrollToRight = useCallback((forceScroll = false) => {
  if (!chartRef.current || data.length === 0) return
  const lastTime = data[data.length - 1].time as number

  // Skip if same timestamp (avoid redundant redraws)
  if (!forceScroll && lastTime === lastScrolledTimeRef.current) return

  const timeScale = chartRef.current.timeScale()
  const rangeStart = lastTime - VISIBLE_RANGE_SECONDS
  timeScale.setVisibleRange({
    from: rangeStart as UTCTimestamp,
    to: lastTime as UTCTimestamp,
  })
  lastScrolledTimeRef.current = lastTime
}, [data])
```

### Part 2: requestAnimationFrame Batching

Batch all chart operations within browser's render cycle (60fps):

```typescript
const rafIdRef = useRef<number | null>(null)

useEffect(() => {
  if (!seriesRef.current || data.length === 0) return

  // Cancel pending frame
  if (rafIdRef.current !== null) {
    cancelAnimationFrame(rafIdRef.current)
  }

  // Batch with browser render cycle
  rafIdRef.current = requestAnimationFrame(() => {
    // ... chart update logic
  })

  return () => {
    if (rafIdRef.current !== null) {
      cancelAnimationFrame(rafIdRef.current)
    }
  }
}, [data, scrollToRight])
```

## Known Issues & Fixes

### Bug: "Cannot update oldest data" error (Fixed 2026-03-19)

**Symptom:** After ~5 minutes of streaming (300+ data points), the chart would intermittently display:
`Chart error: Cannot update oldest data, last time=[object Object], new time=[object Object]`

**Root Cause (two issues):**

1. **Data trimming caused series desync.** In `usePriceHistory`, when history reaches `MAX_HISTORY_POINTS` (300), the oldest point is trimmed via `prev.slice(1)` and a new point is appended — keeping array length at 300. In `price-chart.tsx`, the update logic compared `data.length` vs `prevLength` to decide between `setData()` (full load) and `update()` (incremental). When trim+add kept the length unchanged, the code called `series.update()` with only the latest point. However, the lightweight-charts series internally still held the old untrimmed dataset, so the new point's timestamp could conflict with the series' internal state, triggering the "Cannot update oldest data" error.

2. **Overly aggressive error handling.** Any error in the data update `catch` block called `setChartError()`, which replaced the entire chart DOM with an error `<div>`. This destroyed the chart instance and container element, making recovery impossible without a full component remount.

**Fix applied to `web/src/components/trading/price-chart.tsx`:**

- **Change A — Data update logic:** Replaced the `data.length === prevLength` branch (which called `update()`) with a fallback `else` branch that calls `setData(data)` to fully resync the series whenever trimming occurs (same length) or data length decreases.
- **Change B — Error recovery:** Replaced `setChartError(...)` in the data update catch block with `console.warn(...)` and `dataLengthRef.current = 0`. This keeps the chart alive on transient errors, and the zeroed ref triggers a full `setData()` on the next render cycle, making the chart self-healing.
